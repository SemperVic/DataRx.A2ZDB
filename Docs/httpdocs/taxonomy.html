A2ZDB Basic Taxonomy Modeling

The A2Z object model is first and foremost designed from an enterprise perspective that is a holistic top-down logical model. What this means is the model is created in an abstract manner using taxonomy namespaces to isolate and partition a variety of schemas. Accordingly, a specific application may not implement all the entities in a physical model.

Modeling Overview

A2ZDB is different than a traditional "View Integration Model" or bottom-up relational model in a variety of ways. Most notable is the vast and growing business computing taxonomy infrastructure combined with managed metadata repository dependencies that allows the A2Z object model to be very maliable with extremely flexible definition integration patterns that can be predictably asserted. What this ultimately means is, refactoring storage structure (adding a table or a new column) of your business data becomes a very rare event in feature driven development. Originally - the A2ZDB consisted of 23 EAV Tables, that was almost 20 years ago. Today, the core database architecture requires less than 50 tables. Less than 100% growth in tables in 20 years says something toward the legitamcy of flexibility in this model.

Top-Down Object Model

Father of All Object Models

  • Everything is an "ENTITY"
  • All "ENTITY" data has a "TYPE_OF_ENTITY"
  • An "ENTITY" can have a "RELATIONSHIP" with another "ENTITY"
  • All "RELATIONSHIPS" have a "TYPE_OF_RELATIONSHIP"
  • An "ENTITY" can have one or more "ATTRIBUTES"
  • An "ATTRIBUTE" has a "TYPE_OF_ATTRIBUTE"
  • An "ATTRIBUTE" can have one or more "DATA_VALUES"
Reference: EAV Modeling

A2ZDB makes maximum use of effeciency and performance when designing a taxonomy namespace. Great care is given toward properly identifying the nature of not just the data being stored, but also the relationships within a domain of discourse this model facilitates. Decades of effort has made this model quite complete in it's core form. More over, it is my experience once developers learn and undertsand this holistic approach, they prefer developing under EAV based Semantic models over "View Integration Models". After getting used to develolping under an A2Z taxonomy, you will quickly realize how quickley you can prototype the ground work for your next software project. After your understanding of Arizona and the A2Z modeling strategy has matured, you will start to realize you model much less and your development effort begins at cunsumption of the appropriate taxonomy. When properly implemented, you will be in a "sweet spot" position where your sustainability is greatly increased and your development effort is more focused toward feature enhancements via API libraries and existing taxonomies. In essence, You are no longer writing programs but assembling API's and connecting them to user interfaces with complete separation of concern between the variety of tiers found in modern web tier containers.

View Integration Model

This data representation is analogous to space-efficient methods of storing a sparse matrix, where only non-empty values are stored. In an EAV data model, each attribute-value pair is a fact describing an entity referred to as a "ENTITY" in the above diagram, and a row in an EAV table stores a single fact. The benifit of this model is this technique makes it easy to store any sparse matrix or composite object to the model maximizing code reuse. The con of this strategy is that it requires you to manage relationships via metadata in-place of a database constraint as is the case in a bottom-up model. Ultimately this creates an abstract nature to the naming convention and normalization techniques combined with relationals forms compareds to a constrained relational object model as typically demonstratred with view integration models.

EAV tables are often described as "long and skinny": "long" refers to the number of rows, "skinny" to the few columns.
